# Quick Crypt — PRF / Client-Side userCred Master Plan

> This is the **master plan** spanning all phases. A separate, detailed Phase-1 execution plan will follow.
>
> Names below (modules, fields, headers, functions) are **provisional** — each implementation phase will rethink variable/function names for clarity before coding.

---

## Context — why we are doing this

Today `userCred` (a 32-byte secret, `USERCRED_BYTES=32`) is **generated by the server** (KMS `GenerateRandom`) at first passkey registration, stored KMS-encrypted at rest as `Users.userCredEnc` (+ `userCredEncOld` backup), and **returned in plaintext to the client at login**. The client uses it for three sensitive cryptographic functions in `libs/crypto/src/lib/keys.ts`:

1. Deriving the **hint** cipher key (`_genHintCipherKeyAndIV`).
2. Deriving the per-block **MAC "signing" key** (`_genSigningKey`).
3. As **extra PBKDF2 input** for the primary cipher key (`_genCipherKey`).

Because the server holds `userCred`, it can in principle **reconstruct the hint key and the MAC signing key** (both derive solely from `userCred` plus values the server can see). It **cannot** reconstruct the **primary data cipher key**: that also mixes in a **user-supplied password the server never sees**, so `userCred` is only a defense-in-depth contributor there. The goal of this work is to make the **server never learn plaintext `userCred`** — closing the hint/MAC exposure entirely and strengthening the primary key — by having the client generate `userCred`, encrypt it with a **WebAuthn PRF** output, and store only client-encrypted copies on the server. Hardware-binding the root secret is the strongest realistic mitigation for the long-term-secret threat (true forward secrecy for permanently-persisted offline-decryptable files is not achievable; we are not pursuing DEK/KEK envelope encryption).

Two realities force a **long dual-mode transition**:

- **PRF is not universally available.** A 2026-05 production audit found ~55% of passkeys proven PRF-capable, ~4% a hard "no" floor, ~22% unclassifiable. Both pre-PRF (server-side) and post-PRF (client-side) `userCred` paths must coexist for a long time.
- **We also want stronger authorization.** A past bug let User B trick the server into attaching B's passkey to User A's account. If every sensitive API call required **proof of `userCred` possession**, that bug's blast radius would shrink dramatically — and post-PRF, even a hijacked session can't prove `userCred` it doesn't hold.

**Design tenets (locked with product owner):**

- **Each phase is a stand-alone rollout.**
- An account is **entirely pre-PRF or entirely post-PRF** — never mixed. A `userCred` value never changes once created; recovery and PRF migration preserve it.
- Migration to post-PRF behaves **like account recovery**: a new PRF-capable passkey is created first, and old passkeys are only **soft-deleted (marked deleted)**, hard-deleted on the **next successful login assertion** — never delete the old passkeys before the new PRF passkey is confirmed working.
- Post-PRF, the server holds **N opaque client-encrypted `userCred` blobs** = (one per passkey, encrypted under that passkey's PRF output) **+ 1** in a **recovery record** (encrypted under the 32-byte recovery secret = 16 random ‖ 16 userId). The **only** way to put `userCred` on a new passkey is to decrypt it from an existing passkey's PRF output or from the recovery words.
- **Proof of `userCred`** is post-quantum: a stable keypair **deterministically derived from `userCred`**, public key stored server-side, private key recreated client-side each session. The **same generic proof primitive** is reused to prove knowledge of the recovery secret without the server storing a replayable hash.

## Locked / provisional design decisions (from design discussion)

| Decision | Choice |
|---|---|
| **Signature family** | **ML-DSA (FIPS 204), post-quantum.** Locked as the family. The **library** (libcrux compiled to WASM vs `@noble/post-quantum` pure-JS) and the **parameter set** (44 / 65 / 87) are **decided by Phase-1a head-to-head perf testing** — current leaning **ML-DSA-65**, and libcrux only if its formal-verification benefit justifies the WASM complexity over @noble. Deterministic keygen from a 32-byte seed; FIPS context string for domain separation. |
| **Why PQ here** | The proof **public key is long-lived** (account lifetime, derived from the stable `userCred`), so a future quantum adversary could harvest it and **forge proofs later**. Forging never reveals `userCred` or decrypts files (those rest on quantum-safe symmetric crypto), so the stake is "bypass the authorization layer," not data confidentiality. Since proof is an **AND factor** with the session, PQ genuinely raises the forgery bar — a true independent hardening once post-PRF. |
| **Enforcement scope** | Proof on **every `authorize:true` endpoint, including `getSession`** (refined in the Phase 1 detailed plan — the proof message excludes the CSRF token, so `getSession` can require proof before issuing a CSRF, gating CSRF issuance on `userCred` possession). Per-request cost is validated by the Phase-1a benchmark. |
| **Anti-replay construction** | **Stateless request-bound signature** — sign `H(method ‖ path ‖ timestamp ‖ bodyHash)` (CSRF is **not** in the signed message; the proof is account-bound via the stored pubkey + cookie identity → no cross-account replay). **No nonce table, no extra round-trip.** Same-account replay bounded by a short timestamp window. Per-request cost validated by Phase 1a. |
| **No version anywhere** | Proof requests carry **no version byte**, and **no version is stored** — a derivation change just re-runs the backfill migration to overwrite all pubkeys; a mismatch forces re-login. We never add logic to gracefully handle multiple live versions. |
| **Pubkey provisioning** | **No `hasProofKey` flag.** Every account has a proof pubkey: a one-time **server migration derives + stores the pubkey for all existing accounts** (server still holds `userCred` at rest pre-PRF), and new accounts derive it **server-side at creation**. Post-PRF (Phase 3) the client supplies the pubkey at creation, since the server no longer holds `userCred`. The existing Ed25519-sized `CERT_*` consts are **superseded** (ML-DSA keys don't fit them). |
| **Session keys post-PRF** | `getSessionKey` currently mixes in `userCredEnc`, which is gone for post-PRF accounts. Replace it with **`lastCredentialId`** (already on `Users`) as the per-user/per-session material, applied **uniformly to all accounts** — the secret root remains `jwtMaterial`. No new field needed. Proof code is unaffected. |
| **Recovery proof (Phase 2)** | Client signs a server-issued **challenge** to start recovery. Reuse the existing **`Challenges`** table with a **new purpose** (auto-expires). Recovery proof is **more sensitive** than userCred proof (it authorizes account takeover) and infrequent, so **extra round-trips and a new endpoint are acceptable**. |
| **Master plan location** | `vibes/prf-implementation-plan.md`. |

## End-state architecture (post-PRF)

- **userCred origin:** generated client-side at account creation (post-PRF) or migration; never sent to the server in plaintext.
- **Storage:** N per-passkey blobs (`Authenticators.userCredEnc`, encrypted under each passkey's PRF output) + 1 recovery record (encrypted under recovery-words secret). Server cannot decrypt any of them.
- **Authorization:** session cookie (server-minted, keyed on `lastCredentialId ‖ jwtMaterial`) proves "a session was issued"; the **ML-DSA proof** proves "client holds `userCred`." Deliberately separated in Phase 1 so the post-PRF change is localized.
- **Recovery:** server stores only a **recovery public key** (ML-DSA, derived from the recovery secret) — never the recovery ID. Caller proves knowledge by signing a `Challenges`-table challenge. Recovery record additionally carries the recovery-encrypted `userCred` (post-PRF only).
- **Propagation:** a new passkey gets `userCred` only by decrypting an existing passkey's blob or the recovery record — enforced client-side, structurally impossible for the server to shortcut.

## Phase 1 — Proof of userCred (stand-alone rollout, pre-PRF)

**Goal:** Add an ML-DSA proof-of-`userCred` as additional authorization on (almost) every API call, for both pre- and post-PRF accounts. Sets the stage for PRF while delivering a security win now.

**1a. Throw-away benchmark app (do first; gates primitive, parameter set, and enforcement scope):**
- A **separate throw-away application** that mirrors the main app's technical choices (TypeScript, ESBuild, vitest; runnable as a browser or node app) so we learn the integration and can **repurpose the chosen wrapper code** into the real apps.
- Wire up **both** candidate libraries: **libcrux compiled to WASM** (build via wasm-pack; the repo's `wasm-demo` + `wasm-bindgen` optional dep are the starting point) **and `@noble/post-quantum`** (pure JS), for a **head-to-head** comparison.
- Benchmark **keygen / sign / verify** for **ML-DSA-44, ML-DSA-65, ML-DSA-87** in browser-WASM and node, plus key/sig sizes and the per-request wire overhead. Verify **deterministic keygen from a 32-byte seed** for the chosen library.
- **Outcome decides:** library (if libcrux's formal-verification edge doesn't beat @noble by enough to justify WASM complexity, use @noble), parameter set (leaning 65), and whether "proof on every endpoint" is feasible vs. a narrower scope.

**1b. Generic proof primitive (new module in `libs/crypto`):**
- Keep the crypto lib **focused on crypto**, not app logic. The module is a **generic proof-of-secret-ownership** primitive: derive a deterministic ML-DSA keypair from **any** 32-byte secret + a caller-supplied domain tag, plus `sign`/`verify`. It must serve **both** the userCred proof and the recovery-word proof — nothing userCred-specific lives here.
- `seed = BLAKE2b-256(domainTag ‖ secret)` via libsodium `crypto_generichash` (domain-separated so it can never collide with the cipher-key derivations in `keys.ts`); `keygen(seed)` deterministic. Exported via `libs/crypto/src/index.ts`. Pure, gated on a WASM `ready`/init pattern like libsodium-wrappers.
- App-specific wiring (which secret, which domain, which message) lives in the web app / server, not the lib.

**1c. Data model + provisioning:**
- `Users`: add a proof **public key** field (length-agnostic base64url, plaintext). **No version field.**
- **One-time server migration** (replacing the temporary `postMunge` body) derives + stores the pubkey for **all existing accounts** (server holds `userCred` at rest pre-PRF, so it derives the keypair locally using the shared `libs/crypto` primitive and keeps only the public key). New accounts derive it **server-side at creation**. ⇒ no client registration round-trip and **no `hasProofKey`** in `libs/api`.
- `apps/server/src/consts.ts`: `PROOF_*` constants (skew window, enforce flag, domain tag). Leave `CERT_*` superseded.
- `apps/server/src/urls.ts`: add `requireProof?: boolean` route flag (true for **every** `authorize:true` route, **including `getSession`**).

> See **`vibes/prf-phase1-plan.md`** for the full Phase 1 execution plan (proof contract, ordered steps, rollout).

**1d. Server enforcement (`server.ts`, may import `libs/crypto`):**
- `verifyProof(verifiedUser, httpDetails, headers)`: rebuild `proofMsg`, check timestamp window, ML-DSA-verify against the stored pubkey. Throws `AuthError` on failure. No version negotiation.
- `handler()`: after cookie/CSRF, if `requireProof` and enforce-flag on → enforce; if enforce off → **observe-only** (verify + log, never reject) for safe rollout.

**1e. Web client (`authenticator.service.ts`):**
- Derive + cache the proof **private key** in memory at login (from `getUserCred()` via the generic primitive, then wipe `userCred`); clear it wherever `_csrf` is cleared (logout/forgetUser/cross-tab).
- `_doFetch`: for proof-required calls, reuse the existing body hash, build `proofMsg`, sign, add headers `x-proof-sig` / `x-proof-ts` (no version header) alongside `x-csrf-token`. Client always signs (every account has a pubkey) — no conditional.
- Fresh-tab flow preserved: `getSession` (no proof) mints CSRF; the tab obtains `userCred` via BroadcastChannel (or re-auths), then signs subsequent calls.

**1f. Rollout:** run the server migration → ship client signing in **observe-only** (enforce-flag off) → confirm clean verification in telemetry as clients refresh → flip to **enforce**.

**1g. Tests:** unit (generic-proof spec: determinism, domain separation vs `keys.ts`, sign/verify round-trip, tamper rejection); server (`apps/server/spec`: accept valid, reject expired ts / wrong method-path-body / wrong CSRF / wrong key; migration derives correct pubkey; observe-only never rejects); e2e (`apps/web/tests`: mutate succeeds with proof, enforce rejects tampered/missing proof, multi-tab logout wipes key, `getSession` tab-resume still works).

## Phase 2 — Recovery words without a server-stored recovery ID (stand-alone; pre- and post-PRF)

**Goal:** Server no longer stores the recovery ID; caller proves knowledge of the recovery words without a replayable hash. First release carries **no** client-encrypted `userCred` in the recovery record (no post-PRF accounts exist yet).

**Approach:** Reuse the Phase-1 **generic proof primitive** with a distinct domain tag. Recovery secret = 16 random ‖ 16 userId → ML-DSA keypair; server stores only the **recovery public key** (replacing `recoveryIdEnc` semantics). To recover, the client **signs a challenge issued from the existing `Challenges` table under a new purpose** (auto-expiring); extra round-trips / a dedicated endpoint are fine given infrequency and higher sensitivity. The account is located via the userId embedded in the recovery words. **Change-recovery-words:** an authenticated user (already holding `userCred` from that session) generates new recovery words → new keypair → uploads the new public key, **replacing** the prior recovery record. The server can no longer display recovery words after creation — UI reflects this. Detailed plan authored when Phase 1 completes.

## Phase 3 — New accounts default to PRF (stand-alone; when passkey supports PRF)

**Goal:** New account creation generates `userCred` **client-side** and stores it PRF-encrypted if the registering passkey supports PRF; otherwise the user chooses to switch passkeys or continue pre-PRF (server-side) `userCred`.

**Approach:** Wire the WebAuthn **PRF extension** into `startRegistration`/`startAuthentication` (`prf: { eval: { first: <salt> } }`; note PRF output often needs a follow-up assertion to read on some platforms). Client generates `userCred`, encrypts it under the PRF output (reuse the existing cipher service / `MasterKeyKeyProvider` pattern), stores the per-passkey blob (`Authenticators.userCredEnc`) and the recovery record (now carrying recovery-encrypted `userCred`), and supplies the **proof pubkey** at creation (server can't derive it post-PRF). Set account mode = post-PRF (new `Users` flag). **Rework `getSessionKey`** off `userCredEnc` → `lastCredentialId` (for all accounts). Detailed plan authored when Phase 2 completes.

---

## Phase 4 — Migrate existing pre-PRF accounts to PRF (stand-alone)

**Goal:** Let pre-PRF users on PRF-capable passkeys migrate, mirroring the recovery-link→recovery-words migration-prompt UX in the credentials panel.

**Approach:** Detect PRF support by **attempting PRF during login assertion**; if it succeeds and the account is pre-PRF, show a migration prompt. Migration = recovery-like: create a new PRF-aware passkey, obtain `userCred`, re-encrypt it under the new PRF output + recovery secret, write the per-passkey blob + recovery record, flip the account to post-PRF, **soft-delete** old non-PRF passkeys, and **hard-delete only on the next successful login assertion** (never before the new passkey is confirmed). Needs an `Authenticators` soft-delete marker (e.g., `pendingDelete`/`supersededAt`). Detailed plan authored when Phase 3 completes.

---

## Phase 5 — Deprecate new pre-PRF accounts (stand-alone; future)

Forbid new account creation with non-PRF passkeys; prompt users who log in with non-PRF passkeys to obtain a PRF-capable passkey and migrate. Honors the long fallback period — existing pre-PRF accounts **must keep working** (do not lock out the no-PRF/unclassifiable buckets). Detailed plan authored later.

## Cross-cutting concerns

- **Dual-mode coexistence:** pre- and post-PRF paths must both work through **Phases 1–5** (existing pre-PRF accounts keep functioning even after Phase 5 stops new ones). An account-mode flag gates PRF behavior; an enforce flag gates proof enforcement.
- **Key-rotation invariant:** the proof pubkey and `userCred` change together or not at all. Any future userCred-changing flow must re-derive/re-store the proof pubkey in the same transaction; `proofKeyVersion` (server-only) lets us detect a derivation change and force re-login.
- **No replayable secrets:** never store a hash/KDF of the recovery ID or `userCred` that a thief could replay — always public-key challenge proofs.
- **Clock skew:** short window (env-tunable, ~60 s); optionally correct via a server-time offset only if telemetry shows skew rejections. Never widen the window to mask skew (it directly enlarges the replay window).
- **Naming:** module/field/header/function names here are provisional; refine for clarity at each implementation phase.
- **Secrets hygiene:** never log `userCred`, seeds, or private keys.

## Verification (per phase, end-to-end)

- **Phase 1a perf:** the throw-away app's benchmark numbers (libcrux-WASM vs @noble; 44/65/87; browser + node) are the gate for library, parameter set, anti-replay, and enforcement scope.
- **Build/lint:** use project scripts (`pnpm build:web`, `pnpm build:server`) — never bare nx (SRI/CSP flags). Any WASM bundle must pass strict-CSP on the deployed server (validate-build / `deploy:web validate`).
- **Unit:** full `pnpm test` after any `libs/` change (crypto + server + web + cli), not just the edited project.
- **E2E:** `pnpm serve` + `pnpm test:e2e` (append `--reporter=list` in non-TTY); honor the user-tracking contract and `@nukeall` leak alarm. Add migration/recovery/proof e2e specs alongside `lifecycle.spec.ts`.
- **Rollout (Phase 1):** server migration completes → observe-only telemetry (proof-verified vs proof-failed) as clients refresh → flip enforce.
